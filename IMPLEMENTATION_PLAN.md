# Helpy 项目实施计划（更新版）

## 1. 文档目的

本计划用于在现有项目计划框架基础上，形成一份可执行、可度量、可验收、可控风险的更新版项目实施文档，指导 Helpy 桌面端与 Chrome 扩展端的优化、联调、测试与发布工作。

本次更新重点补充以下内容：

- 梳理现有计划的缺失环节与治理短板。
- 细化阶段划分、关键任务、依赖关系与里程碑节点。
- 明确资源分配清单、职责边界与协作机制。
- 为各阶段和关键任务补充交付标准与验收条件。
- 建立进度跟踪、问题升级、沟通同步与变更管理机制。
- 完善风险识别、预警触发条件、应对预案与回退策略。

## 2. 项目背景

Helpy 当前由两部分组成：

- Electron 桌面应用：负责任务管理、提醒、计时器、标签页展示及扩展连接状态展示。
- Chrome 扩展：负责浏览器标签页追踪、闲置检测、通知提醒，以及与桌面应用通过本地桥接进行通信。

当前计划已覆盖范围、目标、基础阶段与测试方向，但仍存在以下不足：

- 缺少对计划缺口的显式梳理，难以判断扩展内容的必要性和优先级。
- 缺少明确的里程碑定义、阶段出口标准与审批节点。
- 缺少任务级交付标准和验收条件，执行结果难以统一判断。
- 缺少资源配置与责任分工说明，协作边界不清。
- 缺少例行进度跟踪机制、状态口径和偏差纠正方式。
- 缺少沟通协作流程、问题升级路径和变更控制规则。
- 风险项已有基础表述，但缺少触发信号、预防动作与应急预案。

## 3. 项目目标

### 3.1 总体目标

- 在不改变现有 Electron + Chrome Extension 技术路线的前提下，完成 Helpy 现有功能的打磨优化与个性化称呼能力建设。
- 提升桌面端与扩展端的可用性、一致性、性能稳定性及错误可见性。
- 建立统一的显示名称能力，使系统可以在桌面应用、提醒通知、扩展弹窗与设置页面中按统一规则使用用户称呼。
- 在开发完成前形成可复用的测试、验收和发布准备机制。

### 3.2 量化目标

- 关键主流程通过率达到 100%：任务新增、编辑、完成、删除、筛选、提醒触发、标签追踪、扩展连接、个性化显示。
- 关键缺陷清零：发布前无 Severity 1、Severity 2 缺陷遗留。
- 个性化文案覆盖率达到 95% 以上的目标触点。
- 相比当前基线，降低高频轮询带来的无效刷新或不必要同步行为。
- 全量阶段交付均具备明确输出物与验收记录。

## 4. 范围说明

### 4.1 范围内

- 桌面应用界面、交互、状态反馈与性能优化。
- 扩展弹窗与选项页的界面、反馈与性能优化。
- 桌面端与扩展端的显示名称存储、读取、同步与兜底策略。
- 标签追踪、闲置提醒、连接状态与错误提示的优化。
- 自动化测试补充、手工测试、兼容性验证与用户验收。

### 4.2 范围外

- 架构重构为其他平台或迁移到 Web 应用。
- 新增云同步、账号体系、远程存储能力。
- 增加与当前任务管理、提醒、标签追踪无直接关系的新模块。
- 超出本次优化目标之外的大规模安全重构。

## 5. 实施原则

- 先定义规则，再实施功能，避免个性化能力和状态逻辑在两端分叉。
- 先夯实基础层，再展开界面层和联动层改造，降低返工。
- UI 调整与行为逻辑调整尽量分离，便于定位回归问题。
- 每阶段必须具备明确入口条件、输出物和退出条件。
- 所有高风险改动必须配套回退方案、验证方案和问题登记。

## 6. 前提假设

- 目标运行环境以 Windows 桌面端为主。
- 浏览器扩展以 Chromium / Chrome 为主目标环境。
- “称呼功能”定义为以用户可配置显示名称进行友好个性化展示，不包含内容审核或情绪识别能力。
- 用户显示名称支持为空；为空时系统必须稳定回退到默认文案。
- 桌面应用和扩展仍通过现有本地桥接机制通信。

## 8. 阶段划分与执行策略

整体计划建议工期为 8 至 10 个工作日，其中 8 个工作日为标准执行周期，额外 2 个工作日作为缓冲与风险吸收窗口。

### 阶段 0：计划确认与基线对齐

- 建议时长：0.5 天
- 目标：统一范围、称呼规则、验证边界、优先级与产出要求。
- 入口条件：已有现有计划草案、现状评估资料可供审阅。
- 核心任务：
  - 确认显示名称字段命名、默认文案与适用触点。
  - 确认桌面端与扩展端的支持环境、测试边界与发布口径。
  - 确认计划基线、版本节奏、评审机制与验收方式。
- 交付物：
  - 更新版实施计划文档。
  - 个性化触点清单。
  - 测试范围基线。
- 交付标准：
  - 所有阶段有明确依赖、时长、输出物和退出条件。
  - 个性化触点清单覆盖桌面端、通知、扩展弹窗、选项页、状态文案。
- 验收条件：
  - 计划文档经项目负责人确认可执行。
  - 关键术语、边界和优先级无分歧。

### 阶段 1：共享设置与个性化基础能力建设

- 建议时长：1 天
- 目标：建立 `displayName` 的统一数据模型、读写接口、兜底规则和校验机制。
- 入口条件：阶段 0 完成并确认显示名称策略。
- 核心任务：
  - 桌面端新增持久化设置模型，支持显示名称的保存和读取。
  - 提供渲染层可用的 IPC 接口与初始化加载流程。
  - 扩展端补齐设置存储或同步机制，确保两端策略一致。
  - 实现统一格式化辅助函数，处理空值、空白字符、长度限制和默认称呼。
- 交付物：
  - 共享个性化数据契约。
  - 桌面端与扩展端的名称读写能力。
  - 默认文案与兜底规则说明。
- 交付标准：
  - `displayName` 输入校验规则明确且两端一致。
  - 未设置名称时不会出现空白、断句异常或显示 `"undefined"`。
  - 可在应用重启、扩展重开后稳定读取最近一次有效设置。
- 验收条件：
  - 名称保存、读取、更新、为空回退四类场景均验证通过。
  - 两端至少一处 UI 成功展示同一规则下的个性化文案。

### 阶段 2：桌面应用侧体验与性能优化

- 建议时长：2 天
- 目标：提升 Electron 应用的视觉一致性、任务操作体验、状态反馈清晰度和运行效率。
- 入口条件：阶段 1 完成，桌面端可正常读取个性化设置。
- 核心任务：
  - 优化主界面布局、间距、按钮层级、卡片样式与信息密度。
  - 增加显示名称配置入口并提升设置可发现性。
  - 优化空状态、连接状态、错误状态与引导文案。
  - 改善任务创建、编辑、筛选、计时器操作和焦点管理体验。
  - 优化标签追踪显示和状态识别。
  - 识别并减少不必要的轮询、重渲染和高频 IPC。
  - 在提醒、标题、任务反馈等触点接入个性化称呼。
- 交付物：
  - 更新后的桌面端界面与文案系统。
  - 稳定的设置入口与显示名称渲染逻辑。
  - 优化后的状态提示、连接反馈和任务交互体验。
- 交付标准：
  - 主流程无阻断，界面层级清晰，关键按钮可见性提升。
  - 错误提示可读、可理解、可恢复。
  - 个性化文案在桌面端目标触点按规则显示。
- 验收条件：
  - 任务 CRUD、筛选、计时器、提醒、连接状态验证通过。
  - 桌面端视觉与交互评审通过，无阻断性体验问题。

### 阶段 3：扩展侧体验与性能优化

- 建议时长：2 天
- 目标：提升扩展弹窗、选项页和后台逻辑的清晰度、性能与通知有效性。
- 入口条件：阶段 1 完成，扩展端已具备名称读写或同步能力。
- 核心任务：
  - 重构弹窗与选项页的信息布局与状态表达。
  - 优化暂停、运行中、未连接、异常等状态文案与提示逻辑。
  - 减少 1 秒轮询刷新等低效更新方式，优先采用事件驱动或低频同步。
  - 优化闲置检测、通知触发与设置反馈。
  - 将个性化称呼接入弹窗状态文本、通知文案和测试提示。
- 交付物：
  - 更新后的扩展弹窗与选项页。
  - 调整后的后台同步与通知策略。
  - 个性化扩展文案支持。
- 交付标准：
  - 扩展状态可被普通用户快速理解。
  - 通知触达上下文合理，不产生明显干扰。
  - 扩展设置保存后反馈明确、错误提示清晰。
- 验收条件：
  - 暂停、恢复、空闲检测、标签同步、通知发送验证通过。
  - 弹窗与设置页文案一致、状态一致、保存反馈一致。

### 阶段 4：桌面端与扩展端集成加固

- 建议时长：1 天
- 目标：确保双端在名称同步、状态同步、启动顺序和异常恢复方面表现稳定。
- 入口条件：阶段 2 和阶段 3 核心开发完成。
- 核心任务：
  - 确认显示名称的唯一事实来源和冲突处理规则。
  - 验证桥接层对缺失数据、陈旧数据、部分失败的处理策略。
  - 验证 app-first、extension-first、断连、重连、睡眠恢复等场景。
  - 统一双端连接状态、同步状态和错误状态口径。
- 交付物：
  - 同步规则说明。
  - 集成联调记录。
  - 关键异常场景验证结果。
- 交付标准：
  - 双端状态口径一致，用户无需手动执行复杂刷新动作。
  - 启动顺序变化不影响关键功能可用性。
- 验收条件：
  - 集成联调清单全部通过。
  - 名称变更可在预期时间内传播到目标触点。

### 阶段 5：测试、稳定性修复与发布准备

- 建议时长：1.5 天
- 目标：补足自动化覆盖、完成手工与兼容性验证，并形成发布决策依据。
- 入口条件：阶段 4 完成且核心回归问题已关闭。
- 核心任务：
  - 补充设置、个性化、后台逻辑、通知判定和状态辅助逻辑测试。
  - 执行功能测试、兼容性测试、启动顺序测试和回归测试。
  - 完成用户验收清单、已知问题清单和发布建议。
- 交付物：
  - 自动化测试结果。
  - 手工测试清单与结果。
  - UAT 结论与发布准备报告。
- 交付标准：
  - 自动化测试通过，关键回归路径覆盖完成。
  - 手工测试记录完整，问题闭环状态清晰。
  - 发布建议包含是否可发布、遗留问题与规避说明。
- 验收条件：
  - 无阻断级缺陷遗留。
  - 功能、兼容性、体验和稳定性结论均达到发布门槛。

### 11.3 单人执行模式说明

如果项目由单人开发执行，则角色可合并为“项目负责人 + 全栈开发 + 测试执行”模式，但需同步调整以下策略：

- 阶段 2 与阶段 3 不并行，以串行为主。
- 每日预留至少 0.5 小时用于回归验证和问题记录。
- 缓冲时间建议从 2 天提高到 3 天。

## 13. 依赖关系与关键路径

### 13.1 关键依赖

- 显示名称的数据契约必须先确定，否则双端文案和设置入口无法稳定落地。
- 桌面端和扩展端的兜底文案规则必须先统一，否则容易出现不同触点表述冲突。
- 集成联调必须在阶段 2 和阶段 3 完成主要开发后进行，否则联调结果无稳定意义。
- 测试阶段依赖各阶段交付达到“可测”标准，否则会造成反复回归和计划漂移。

### 13.2 关键路径

阶段 0 -> 阶段 1 -> 阶段 2 / 阶段 3 -> 阶段 4 -> 阶段 5

其中阶段 1、阶段 4、阶段 5 位于关键路径上，任一延迟都将直接影响整体交付时间。

## 14. 进度跟踪机制

### 14.1 跟踪频率

- 每日站会：15 分钟，更新昨日完成、今日计划、阻塞事项。
- 阶段评审：每个阶段结束时执行，确认是否满足退出条件。
- 里程碑检查：M0 至 M5 节点执行结果审阅和偏差判断。

### 14.2 状态口径

所有任务统一使用以下状态定义：

- 未开始：尚未投入执行。
- 进行中：已启动，尚未达到交付标准。
- 待验证：开发完成，等待测试或评审。
- 已完成：验证通过并有结论记录。
- 阻塞：因依赖、缺陷或决策问题暂时无法推进。

### 14.3 进度度量指标

| 指标           | 计算方式                        | 目标值   | 预警阈值           |
| -------------- | ------------------------------- | -------- | ------------------ |
| 阶段按时完成率 | 按时完成阶段数 / 阶段总数       | 100%     | 低于 90%           |
| 里程碑达成率   | 按计划完成里程碑数 / 里程碑总数 | 100%     | 任一关键里程碑延误 |
| 缺陷关闭率     | 已关闭缺陷数 / 缺陷总数         | 持续提升 | 连续 2 天无改善    |
| 回归通过率     | 通过用例数 / 执行用例数         | 95% 以上 | 低于 90%           |
| 范围变更控制率 | 已审批变更数 / 变更总数         | 100%     | 出现未审批变更     |

### 14.4 偏差处理机制

- 任务延迟超过 0.5 天时，必须在当日站会说明原因和纠偏措施。
- 关键路径任务延迟时，优先调整资源和范围，而不是压缩测试时间。
- 同一问题连续阻塞超过 1 个工作日时，必须升级到项目负责人决策。

## 17. 变更管理机制

### 17.1 变更范围

以下情况视为正式变更：

- 新增原计划未包含的功能点。
- 关键交互逻辑、同步策略或测试范围发生变化。
- 阶段工期变化超过 0.5 天。
- 发布门槛或验收标准发生变化。

### 17.2 变更流程

1. 提出变更申请，说明原因、影响、收益与紧急度。
2. 评估对范围、工期、质量、风险和测试的影响。
3. 项目负责人决定接受、延后或拒绝。
4. 更新计划、任务清单、里程碑和验收口径。
5. 将变更结果同步给所有执行角色。

### 17.3 变更控制原则

- 没有审批记录的变更不得直接进入执行。
- 关键路径上的新增需求默认进入后续迭代，除非有明确业务优先级证明。
- 不以压缩测试时间作为默认赶工手段。

## 18. 测试与验收方案

### 18.1 功能测试

- 验证任务新增、编辑、完成、删除、筛选、提醒与逾期处理。
- 验证计时器启动、暂停、重置、超时提醒和刷新表现。
- 验证扩展暂停、恢复、标签追踪、闲置检测、通知触发和设置持久化。
- 验证显示名称在所有目标触点正确显示，并在未设置时正确回退。

### 18.2 兼容性测试

- Windows 桌面环境下 Electron 应用运行与提醒表现。
- Chrome / Chromium 环境下扩展行为与权限能力。
- 双端启动顺序、重连、休眠恢复、桥接不可用等场景。

### 18.3 用户验收测试

- 以日常任务管理、浏览追踪、提醒触达、称呼展示为核心场景执行验收。
- 验证界面是否更清晰、状态是否更直观、文案是否自然。
- 验证个性化称呼是否“有帮助但不过度打扰”。

### 18.4 发布门槛

- 自动化测试通过。
- 功能测试、兼容性测试、UAT 均有结果记录。
- 无 Severity 1、Severity 2 缺陷遗留。
- 已知问题被记录并附带规避说明，不影响主流程。

## 20. 项目完成定义

满足以下条件时，本次项目可判定为完成：

- 更新版实施计划已确认并作为执行基线使用。
- 各阶段交付物均完成并通过对应验收条件。
- 桌面端与扩展端均完成优化目标，核心功能无阻断。
- 显示名称能力在目标触点稳定生效且存在可靠兜底。
- 联调、测试、用户验收与发布准备工作完成。
- 不存在影响发布的高优先级遗留问题。
